# 项目规则

本项目用于沉淀视觉 AI / 算法面试题卡。后续所有新增题目、答案整理和页面更新，默认遵循以下规则。

## 1. 回答流程

1. 用户先发图片或题目。
2. 助手必须先在对话里输出：
   - 题目识别结果
   - 对应简答
   - 若是手写题，再附代码
3. 在答案已经输出给用户之后，助手再把内容写入项目文件。
4. 不需要等待用户再次确认是否写入，默认直接落库。

一句话总结：

**先答题展示，后写入项目。**

## 2. 数据写入规则

1. 所有题目统一写入 `data.js`。
2. 新增题目默认插入到题库前部，保证最新题目优先显示。
3. 日期使用当前对话当天日期。
4. 如果没有特别说明，新增题目默认归类为：
   - `company: "字节跳动"`
   - `topicGroup: "基础知识点"`

## 3. 题目内容规则

1. 答案以面试口语化、可直接复述为目标，不要过度学术化。
2. 如果题目解释不完整，尤其涉及公式、指标、推导、统计量时，必须先解释基础概念，不要跳步。
3. 如果用户指出“回答太简单”或“没解释清楚某个量”，应直接补足该题并同步更新题库。

## 4. 缩写规则

凡是出现英文缩写，默认同时给出：

1. 英文全称
2. 中文含义
3. 一句简短解释

例如：

- `AUC` = `Area Under the Curve`
- `ROC` = `Receiver Operating Characteristic`
- `TPR` = `True Positive Rate`
- `FPR` = `False Positive Rate`
- `CPM` = `Cost Per Mille`
- `CPC` = `Cost Per Click`
- `ROI` = `Return On Investment`

## 5. 手写题规则

如果题目属于以下类型，除了文字解释，还必须给出代码：

- 手写代码
- 手写卷积
- 手写 BP / 反向传播
- 手写 BN
- 手写 Softmax
- 其他明确要求“实现”的题目

要求：

1. 代码尽量简洁，适合面试时白板 / 编辑器手写。
2. 优先使用 Python / NumPy 风格示例。
3. 页面中答案要以代码块形式展示，不可与普通文本混在一起。

## 6. 公式题规则

1. 涉及公式的问题，优先给出文字解释，再给公式。
2. 页面中公式应尽量使用公式块渲染，而不是普通文本直出。
3. 若题目中出现 `TPR`、`FPR`、`AUC`、`FLOPs` 等量，必须解释它们的定义。

## 7. 页面与交互规则

1. 首页卡片展示摘要。
2. 点击卡片后，在详情层查看完整答案。
3. 页面保留并持续维护以下筛选能力：
   - 公司筛选
   - 专题筛选
   - 日期筛选
4. 公司、专题、日期都可以点击进入对应筛选结果。

## 8. 默认风格规则

1. 页面风格保持简洁、克制、留白充足。
2. UI 方向参考 Claude 风格：
   - 温和米白背景
   - 低对比边框和阴影
   - 信息层级清晰
   - 不使用花哨动画和强烈颜色

## 9. 后续扩展规则

如果后续新增公司或专题，应优先使用以下字段扩展，而不是硬编码页面：

```js
company: "阿里巴巴"
topicGroup: "项目知识点"
```

这样前端筛选逻辑无需再次重构。

## 10. 助手执行承诺

从本规则文件建立后，助手后续默认按本规则执行；若用户当次消息明确提出新的例外要求，则以用户当次要求为准。
